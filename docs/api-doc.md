# API 文件

## 概要

這是一套**個人單一名片系統**：資料庫中只會存在一筆名片資料，由系統預先建立，前端不再有「新增名片」的能力。API 僅提供**讀取**與**更新**這筆資料的動作，且永遠只針對這唯一的一筆。

所有動作皆透過單一 API URL，以 `POST` 傳送 JSON。JSON 中的 `action` 欄位決定當次請求的行為。

> **設計原則**
> - 個人使用：全庫僅一筆名片，不需要識別碼（無 `card_id`）、不需要清單、不需要新增/刪除。
> - JSON 結構不超過兩層；回應將名片欄位直接放在第一層，與 `success` / `message` 並列，方便前端解析。
> - 不記錄建立時間、修改時間（個人名片無版本或稽核需求）。

## 資料模型

資料庫中預先存在**唯一一筆**名片紀錄。API 不負責建立它，只能讀取或更新它的內容。

### 名片欄位

| 欄位 | 型別 | 說明 |
|------|------|------|
| `name` | string | 姓名 |
| `title` | string | 職稱 |
| `company` | string | 服務單位（公司名稱） |
| `tax_id` | string | 公司統編 |
| `phone` | string | 連絡電話 |
| `fax` | string | 傳真號碼 |
| `email` | string | 信箱 |
| `line_id` | string | Line ID |
| `services` | string | 服務項目（以頓號或逗號分隔） |

> 不存在 `card_id`、`created_at`、`updated_at` 欄位。個人名片只有一筆，無需識別與時間標記。

## API 端點

| 項目 | 內容 |
|------|------|
| URL | `https://example.com/api/business-card.php` |
| Method | `POST` |
| Content-Type | `application/json` |

> **備註**：上方 URL 為預留範例，實際上線前請替換為正式伺服器位址。

## 共同請求欄位

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `action` | string | 是 | 動作類型，僅允許 `read` 或 `update`。 |
| `timestamp` | string | 否 | ISO 8601 格式時間戳，僅用於請求記錄與除錯，不寫入資料。 |
| `request_id` | string | 否 | 前端產生的唯一識別碼，可用於追蹤請求。 |

## 動作說明

### read — 讀取名片

- 用途：讀取唯一一筆名片資料。不需帶入任何識別碼。
- 行為：回傳資料庫中預先建立的那筆名片內容。

**請求範例**

```json
{
  "action": "read"
}
```

**成功回應**

```json
{
  "success": true,
  "message": "讀取成功",
  "name": "王小明",
  "title": "業務經理",
  "company": "精準金屬加工股份有限公司",
  "tax_id": "12345678",
  "phone": "02-2345-6789",
  "fax": "02-2345-6790",
  "email": "ming.wang@example.com",
  "line_id": "wang_metal",
  "services": "CNC銑削、車床加工、金屬表面處理、雷射切割"
}
```

**失敗回應**

```json
{
  "success": false,
  "message": "名片資料不存在，請先於資料庫預先建立"
}
```

### update — 更新名片

- 用途：更新唯一一筆名片資料。不需帶入識別碼，後端直接更新該筆紀錄。
- 行為：以請求中的欄位覆寫資料庫中既有的那筆名片。未提供的欄位依實作策略而定（建議整體覆寫，前端送出完整資料）。

**請求範例**

```json
{
  "action": "update",
  "name": "王小明",
  "title": "業務經理",
  "company": "精準金屬加工股份有限公司",
  "tax_id": "12345678",
  "phone": "02-2345-6789",
  "fax": "02-2345-6790",
  "email": "ming.wang@example.com",
  "line_id": "wang_metal",
  "services": "CNC銑削、車床加工、金屬表面處理、雷射切割"
}
```

**成功回應**

```json
{
  "success": true,
  "message": "名片更新成功"
}
```

**失敗回應**

```json
{
  "success": false,
  "message": "更新失敗，名片資料不存在"
}
```

## 通用回應欄位

| 欄位 | 型別 | 說明 |
|------|------|------|
| `success` | boolean | 請求是否成功，`true` 成功、`false` 失敗。 |
| `message` | string | 回應訊息，用於顯示操作結果。 |
| `name` … `services` | string | `read` 成功時直接回傳名片欄位；`update` 成功時不回傳欄位。 |

## 狀態碼

| HTTP 狀態碼 | 說明 |
|-------------|------|
| 200 | 請求成功處理完成。 |
| 400 | 請求格式錯誤、缺少必要欄位，或不支援的 `action`。 |
| 404 | 名片資料不存在（資料庫未預先建立）。 |
| 500 | 伺服器內部錯誤。 |

## 不支援的動作

以下動作在個人單一名片系統中**不提供**，後端應對未知 `action` 回傳 400：

| 動作 | 原因 |
|------|------|
| `create` | 名片由資料庫預先建立，前端不負責新增。 |
| `delete` | 個人名片不刪除，僅更新內容。 |
| `list` | 全庫僅一筆，無清單需求，使用 `read` 即可。 |

## 備註

- 所有動作皆使用 `POST`，統一通訊介面。
- 請求與回應皆為單層 JSON，不超過兩層結構。
- 個人名片無識別碼：後端固定操作預先建立的那一筆，前端不需也不應傳送 `card_id`。
- 不記錄時間戳：欄位集中於名片內容本身，無 `created_at` / `updated_at`。
- 實際 URL 與認證方式待後續補充。
